Stream Refresh Rates

Stream content is updated as follows:

  • Streams, On screen (in view, active user)—2 minutes
  • Streams, On screen (in view, idle user)—15 minutes
  • Streams, Off screen—Every two hours or less, depending on channel.
  • Collections, On screen (Including Inbox)—Requests for new data sent every 2 minutes, but can take longer if there are many streams in the Collection.
  • Recent Engagement and All Activity—Update frequency depends upon the age of the item.

See Content Metrics.

When Streams Are Not “In View” by a User

(Offline—Updating is handled by internally, rather than the user’s browser.) Every 30 minutes we check to see if any streams are “stale” (older than one hour by default, but varies by channel). If there are stale streams that haven’t been fetched either online (meaning through the user’s browser when the stream is in view) or offline (meaning that has asked the channel for fresh data using the API) in the last 12 minutes, we queue fetches for them. So most streams are fetched automatically no more often than once every 30 minutes, but the rate varies by channel. For Facebook and Twitter, the “12 minute” window does not apply; when offline, the system checks every 30 minutes for Facebook and Twitter streams not fetched within the previous minute.

When Streams Are “In View” by a User

(Online—Updating is handled by the user’s browser). When a user is idle (meaning no keyboard/mouse/page input activity for 10 minutes), fetches are done every 15 minutes. When the user is not idle, the streams they have on screen are done every 2 minutes. These fetches work whether the stream is stale or not, effectively bypassing ’s internal fetching system entirely, so long as there's a browser sending requests. (The only exceptions are Facebook and Twitter streams, for which the “stale” time is always honored, and the manual “refresh” button has no effect on data fetching if it is used more than once within the “stale” time.) In these fetches the browser keeps an open connection while its web server accesses the services and returns any new items. These fetches can time out after 5 minutes, but that’s not a problem for a single stream fetch. One exception is Social Mention, however, which we only fetch offline because of the channel’s very slow response times.

A Collection, such as the Inbox, or a user-defined collection, is a group of multiple streams. At least every two minutes (but also when a user clicks Refresh) the browser sends a request to for updated data for all streams in visible collections. Immediately, provides any data it has collected since the last request, and if there are no fetches currently executing, queues up a batch of stream fetches to the channels of all constituent streams to obtain fresh data. Once these fetches are completed, it begins again after two minutes or the next time the user requests an update to the collection.

For both offline fetches and collection fetches, there is some latency associated with queued requests; it can take anywhere from a few seconds to up to 10 minutes for a server to be available. So refresh times could take up to an extra 10 minutes beyond the scheduled “fetch time” to provide new data to the stream.
