Security Settings

If a company administrator makes changes on this screen, an email is automatically sent to inform the other company administrators.

All activity must take place over https; SSL is required.

Password Authentication Settings

Set Password expiration—Set the frequency at which users must change their passwords. Click the Set Password Expiration checkbox, enter a number in the field, and select Months or Days. When a user’s password expires, the user must set a new one the next time log-in is attempted.

Enforce strong passwords—In addition to requirements set within industry standard algorithms for password strength, the following are enforced: minimum of eight alphanumeric characters, at least one special character, at least one number, at least one uppercase letter, at least one lowercase letter, cannot match previous four passwords, and cannot resemble an English word or a simple pattern (for example, 12345678 or 09876543).

Customize CAPTCHA Thereshold—By default, after five unsuccessful log in attempts, a CAPTCHA response is required. The CAPTCHA feature presents an image containing a text string the user must enter correctly before accessing the application. To change the number of attempts permitted, click the checkbox, and set the number of unsuccessful login attempts at which a “CAPTCHA” confirmation is required. The CAPTCHA feature presents an image containing a text string the user must enter correctly before accessing the application.

Enable bad password lockout—This control enables you to define how may attempts users get to log in to and how long they'll be locked out if they fail to log in in the permitted number of attempts. Before a user reaches the maximum number of permitted log-in attempts, will display a warning. Once a user is locked out, red text appears that tells the user that he has been locked out and what to do to reset his password.

(Deprecated) Single Sign-On Using SAML

When SAML is configured, companies can allow (or require) their employees to access with their single sign-on credentials using the Security Assertion Markup Language (SAML) 2.0 standard. The feature relies on a valid certificate from an external identity provider (IDP); three months prior to certificate expiration, sends an email notification to all company administrators.

Single Sign-on and Company Owner
The user designated as the company owner is always able to log in to without the IDP credentials, even when Single Sign-on has been made mandatory for all other users, so that there is always one person who can log in to the system if there is a problem with the provider or certificate. All users with Company Admin privileges can configure the Security tab and Single Sign-on, however.

Once SAML has been set up, administrators can choose to make it optional or mandatory for their users to log in via the IDP. If Single Sign-on is enabled, but not required, users can log in to using either the regular URL or the IDP. If Single Sign-on is required, users must log in through the IDP.

If Single Sign-on using SAML is mandatory, whenever new users are added to the system the invitation email contains the link to log in through the identity provider. Users should bookmark the login URL that contains the required token. If it is enabled, but not required, new users are sent the usual reset password link to access the system.

Similarly, when it is mandatory and (five) login attempts from the IDP fail, the user’s IP address is “locked” and a CAPTCHA confirmation is required. When it is enabled, but not required, and a log-in attempt through the regular URL fails, the user has the number of retries specified in the Password Authentication Settings panel of the Security tab.

Email Assertion Requirements / Format

uses the value of the saml:NameID tag in the saml:Subject to identify the user logging in. The following is an example of a sanitized, valid saml:Subject tag that logs into as test-user@examplecompany.com:

<saml:Subject >
<saml:NameID Format="urn:oasis:names:tc:SAML:2.0:nameid-format:transient" > test-user@examplecompany.com</saml:NameID >
<saml:SubjectConfirmation Method="urn:oasis:names:tc:SAML:2.0:cm:bearer" >
<saml:SubjectConfirmationData NotOnOrAfter="..." Recipient="..." InResponseTo="..."/ >
</saml:SubjectConfirmation >
</saml:Subject >

requires the customer certificate to validate the signature of their identity provider on the assertion. The certificate chain can be accessed by the customer at the transport level via SSL.

Configure Single Sign-on Authentication

  1. From your name > My Company, click the Security tab.
  2. In the Single Sign-On Authentication section, click the Yes button.
  3. Provide the following:
    • Issuer URL
    • Login URL
    • Logout URL
  1. ( Admin only) Select SAML version. Version 2 (default) maps to SAML library version 1.11.0. Version 1 (legacy support) maps to SAML library version 1.8.2.
  2. Click Browse and upload certificate The system checks the validity of the certificate. If the certificate is good, it displays “verified.”
  3. Copy and paste the URLs to complete SAML setup with your IDP:
    • Login URL
    • Logout URL
    • Consumer URL
    • Audience URL
    • Recipient URL
  1. Click Save. The system displays the URLs necessary to implement single sign-on for users.
  2. Complete any necessary testing.
  3. (Optional) Click Make Single Sign On Mandatory to require all users to sign in using the IDP.

Restrict Login by IP

Click the checkbox, and add IP or IP Range (using CIDR notation) that will be allowed to access the application. This is a “white list”; include IPs or ranges that are permitted.

Edit Inactivity Time-out

By default, (when NO is selected) user sessions time out after 4 hours of inactivity. Activity initiated by (such as updating streams in the Inbox) does not count as user activity. To change the inactivity time-out, click Yes and provide new values in the Hours and Minutes fields. Minimum amount of time is 15 minutes; maximum is 8 hours.

Note that time spent logged in to via a third-party app while using the Engage Button counts as user activity.

If a user is a member of multiple companies, the Inactivity and Absolute time-out values for the current company are used. When a user switches to a different company, the timers are set back to zero (minutes), and the new company’s time-out settings are used.

Known Issue with Inactivity Timeouts for Multiple Companies
At 7.0, the ability to honor the timeout values of the user’s current company was implemented. However, the app is enforcing the shortest timeout value among all Companies the user has access to, regardless of which company the user is in. This is a known issue.

Configure Absolute Time-out

User is logged out after the selected amount of time, regardless of user activity or system activity. A warning message appears five minutes before the user is logged out. If an Inactivity timeout has also been set, the Absolute Time-out value must be greater than the Inactivity Timeout value. To set an absolute timeout, click Yes, and provide values in the Hours and Minutes fields. When enabled, the default is 8 hours, and the upper limit is 12 hours.

If the Company requires use of SAML to log-in, Absolute Time-out cannot be set in , because the timeout is managed by the third-party provider.

Expire Passwords Immediately

If you need to require all users to reset their passwords immediately (such as making changes to security settings), click the checkbox, then click Save.
